# Implementation Plan: 日报助手 MVP

**Branch**: `001-desktop-daily-report` | **Date**: 2026-03-24 | **Spec**: [/root/workspace/daily-report-assistant/specs/001-desktop-daily-report/spec.md](/root/workspace/daily-report-assistant/specs/001-desktop-daily-report/spec.md)
**Input**: Feature specification from `/specs/001-desktop-daily-report/spec.md`

## Summary

为 Windows 11 提供一个本地优先的桌面日报助手 MVP，优先打通“全局快捷键快速记录 ->
本地保存 -> 下班前自动弹出汇总 -> 编辑状态与备注 -> AI 润色 -> 生成并复制日报”的最小闭环。
推荐采用 Tauri 2 作为桌面宿主，React + TypeScript 承担界面层，Rust 承担桌面能力、
数据访问和 AI 接入，SQLite 承担本地持久化。该方案能够在保持桌面能力完整的前提下，
控制包体和运行时资源占用，并为后续扩展提醒时间、日报模板、更多状态类型和更多 AI 服务保留清晰边界。

## Technical Context

**Language/Version**: Rust stable 1.7x+、TypeScript 5.x  
**Primary Dependencies**: Tauri 2、React 18、Vite、SQLx 或 rusqlite、serde、chrono  
**Storage**: 本地 SQLite 数据库 + 本地配置存储（同库 settings 表；敏感字段单独做脱敏处理）  
**Testing**: Rust unit tests、TypeScript unit tests、关键流程手动验收脚本  
**Target Platform**: Windows 11 x64 桌面环境  
**Project Type**: 单仓库桌面应用（Tauri 宿主 + Web UI）  
**Performance Goals**: 热启动后快速记录窗口在 300ms 内可交互；单条任务保存反馈在 100ms 级；当日 100 条任务内汇总与生成无明显卡顿  
**Constraints**: 必须离线可记录/查看/编辑/生成基础日报；必须支持全局快捷键、自动提醒、窗口唤起、剪贴板复制；必须避免敏感配置泄露；MVP 不引入账号、云同步与复杂项目管理  
**Scale/Scope**: 单用户、本机使用、每天数十到一百条任务记录、3 个核心窗口、1 条 AI 润色链路

## Recommended Technical Approach

### 1. 推荐技术方案

- **桌面壳/宿主方案**：Tauri 2。
  原因：原生桌面能力覆盖更贴近需求，适合全局快捷键、窗口控制、系统托盘、剪贴板和本地文件/数据库访问；包体与运行资源通常优于 Electron。放弃 Electron 的原因是运行时更重，不利于“低打扰”和常驻后台。放弃 WPF/.NET 的原因是前端开发与跨层复用灵活度较低，后续 UI 迭代和 AI 接入抽象成本更高。
- **前端界面方案**：React + TypeScript + Vite。
  原因：适合快速构建 3 个核心窗口，状态管理清晰，后续扩展模板与设置页也更直接。
- **本地存储方案**：SQLite。
  原因：支持本地优先、结构化查询、按日期聚合、事务保护和后续迁移；比纯 JSON 文件更适合避免覆盖冲突与数据损坏。
- **定时提醒实现方式**：应用常驻时由本地调度器按分钟级轮询当前时间与提醒设置，并在跨日、设置变更、应用恢复时重算当日提醒。
  原因：MVP 目标是应用运行期间的可靠提醒，不引入 Windows 任务计划器等外部依赖。
- **全局快捷键实现方式**：通过 Tauri 宿主层封装全局快捷键注册、冲突检测和重试注册逻辑。
- **剪贴板能力实现方式**：通过 Tauri 宿主层统一封装复制接口和错误反馈，避免界面层直接处理平台差异。
- **AI 接口适配方式**：在 Rust AI 接入层实现 OpenAI 风格 API 适配器，统一处理 Base URL、API Key、模型名、请求体构造、超时和响应解析；界面层只处理“请求润色/展示结果”。

### 2. 架构与模块设计

#### 桌面能力层

- 负责全局快捷键注册、窗口显示/隐藏/聚焦、托盘入口、剪贴板复制、定时提醒调度和应用生命周期事件。
- 对上只暴露稳定命令：
  - `register_hotkey`
  - `show_capture_window`
  - `show_daily_summary_window`
  - `copy_text`
  - `reschedule_reminder`
- 统一处理 Windows 11 环境下的能力失败场景，返回用户可理解的错误类型。

#### 数据存储层

- 负责 SQLite 初始化、表结构迁移、事务读写、按日期查询、配置读写和数据恢复保护。
- 任务数据与应用设置都放在本地数据库中，便于统一迁移和备份。
- 对关键写操作采用事务：
  - 新增任务
  - 批量更新状态/备注
  - 保存设置
- 保留 schema version，后续扩展状态类型、模板配置、更多 AI 服务时通过 migration 演进。

#### 业务逻辑层

- 负责任务新增、状态修改、备注编辑、当日任务聚合、日报内容组装、提醒判断与触发、空日报处理。
- 核心服务建议：
  - `TaskService`
  - `DailyReportService`
  - `ReminderService`
  - `SettingsService`
- 业务层不直接依赖 UI，统一通过输入 DTO 和返回 DTO 交互，便于自动化测试。

#### AI 接入层

- 负责 OpenAI 风格 API 的请求封装、配置校验、prompt 构造、响应解析、超时与异常映射。
- 支持两种输出：
  - `basic_report_text`：本地规则生成
  - `polished_report_text`：AI 润色生成
- 明确降级策略：只要 AI 配置缺失或调用失败，立即回退到基础日报生成，不阻塞用户复制结果。

#### 界面层

- 负责快速记录窗口、当日汇总编辑窗口、设置页/配置窗口、生成结果展示与复制反馈。
- 以窗口和流程为边界组织：
  - `CaptureWindow`
  - `DailySummaryWindow`
  - `SettingsWindow`
- UI 优先展示业务结果，不承载复杂平台逻辑。

## Data Model Design

### 3. 数据模型设计

#### TaskEntry

- **用途**：表示用户某一天的一条任务记录，是快速记录与日报汇总的基础单元。
- **关键字段建议**：
  - `id`: 唯一标识
  - `date`: 所属日期，格式化后可直接用于按日查询
  - `created_at`: 创建时间
  - `content`: 任务标题/内容
  - `status`: 任务状态，MVP 枚举为 `done` / `in_progress`
  - `note`: 备注，允许为空
  - `updated_at`: 最近编辑时间
- **约束**：
  - `content` 非空
  - `status` 只能取枚举值
  - `date` 与本地时区一致

#### DailyReportDraft

- **用途**：表示某一天的日报汇总草稿和生成上下文，供界面显示、AI 润色和结果复制使用。
- **关键字段建议**：
  - `date`: 汇总日期
  - `entries`: 当日任务快照列表
  - `done_entries`: 已完成任务集合
  - `in_progress_entries`: 进行中任务集合
  - `basic_report_text`: 基础日报文本
  - `polished_report_text`: AI 润色后的日报文本，可为空
  - `last_generated_at`: 最近生成时间
  - `generation_source`: `basic` / `ai`
- **用途说明**：
  - 不要求 MVP 持久化整个 draft 表；可由业务层按需从当日任务实时组装。
  - 若后续增加模板切换或多版本草稿，可再演进为持久化对象。

#### AppSettings

- **用途**：保存本地应用配置与 AI 接入配置。
- **关键字段建议**：
  - `reminder_enabled`: 是否启用提醒
  - `off_work_time`: 用户设置的下班时间
  - `remind_before_minutes`: 默认 5
  - `global_hotkey`: 快捷键字符串
  - `ai_enabled`: 是否启用 AI 润色
  - `ai_base_url`: AI 服务地址
  - `ai_model`: 模型名
  - `ai_api_key_ref`: API Key 存储引用或本地受保护字段
  - `report_template`: 当前日报模板标识，MVP 默认单模板
  - `updated_at`: 最近更新时间
- **约束**：
  - 允许 AI 配置为空
  - `off_work_time` 必须可解析
  - 修改提醒设置后必须触发重新调度

## Page and Window Design

### 4. 页面/窗口设计

#### 快速记录窗口

- **职责**：接收用户当前任务输入并立即保存。
- **MVP 内容**：
  - 单个任务输入框
  - 保存按钮
  - 简单反馈信息
- **交互要求**：
  - 快捷键呼出后默认聚焦输入框
  - 回车可保存
  - 保存后可关闭窗口或保留继续录入模式

#### 当日汇总编辑窗口

- **职责**：展示当天全部任务，允许修改状态与备注，触发生成、AI 润色和复制。
- **MVP 内容**：
  - 当日任务列表
  - 每条任务的状态切换与备注输入
  - “生成基础日报”
  - “AI 润色”
  - “复制结果”
  - 结果文本预览区域
- **交互要求**：
  - 进入时默认加载当天所有任务
  - 修改后再次生成必须覆盖旧结果
  - AI 失败后保留基础结果与编辑状态

#### 设置页 / 配置窗口

- **职责**：管理提醒时间、快捷键和 AI 参数。
- **MVP 内容**：
  - 下班时间
  - 提前提醒分钟数
  - 全局快捷键
  - AI 开关
  - Base URL
  - API Key
  - 模型名
- **交互要求**：
  - 保存后立即生效
  - AI 配置可为空
  - 敏感字段回显时要避免直接暴露完整值

#### 托盘或后台常驻入口

- **是否纳入 MVP**：建议纳入 MVP。
- **原因**：应用需要支持全局快捷键和自动提醒，后台常驻入口能显式表达应用仍在运行，并提供打开汇总窗口、打开设置页和退出应用的最低可用控制面板。
- **范围控制**：MVP 仅做最小托盘菜单，不做复杂托盘状态展示。

## Core Flow Design

### 5. 核心流程设计

#### 1. 用户按快捷键记录任务的流程

1. 应用启动后注册全局快捷键。
2. 用户按下快捷键。
3. 桌面能力层显示并聚焦快速记录窗口。
4. 用户输入任务内容并提交。
5. 业务层校验内容非空，构造 `TaskEntry`，默认状态为 `done`。
6. 数据层事务写入本地 SQLite。
7. UI 给出保存成功反馈；若失败则提示失败且不关闭输入内容。

#### 2. 到提醒时间自动弹出汇总窗口的流程

1. 应用启动或设置变更后，`ReminderService` 计算当日提醒时间。
2. 调度器按分钟级检查当前时间是否达到提醒阈值。
3. 命中提醒条件后，触发窗口显示/聚焦。
4. 业务层查询当日全部 `TaskEntry` 并组装 `DailyReportDraft`。
5. UI 展示任务列表；若当日无任务，则展示空状态与补录入口。

#### 3. 用户编辑状态与备注后生成日报的流程

1. 用户在汇总窗口中修改任务状态与备注。
2. 业务层保存变更并刷新当日任务快照。
3. 用户点击“生成基础日报”。
4. `DailyReportService` 按固定模板输出基础日报文本：
   - 已完成
   - 进行中
   - 补充说明（仅有备注时显示）
5. 结果展示区刷新，允许复制。
6. 若用户点击“AI 润色”，则在最新草稿基础上发送到 AI 接入层。

#### 4. AI 润色失败时的降级流程

1. 进入 AI 润色前先校验 `ai_enabled`、Base URL、API Key、模型名。
2. 任一配置缺失时直接返回配置错误，不发请求。
3. 请求超时、网络失败、返回非法内容或解析失败时，统一映射为 `AiPolishFailed`。
4. UI 显示“AI 润色失败，已保留基础日报内容”。
5. 结果区继续展示 `basic_report_text`，用户仍可复制和发送。

#### 5. 用户复制日报的流程

1. 用户在结果区点击复制。
2. UI 调用桌面能力层复制接口。
3. 成功时显示已复制反馈。
4. 失败时显示失败提示，并保留文本内容供用户手动复制。

## Risk and Trade-Offs

### 6. 风险与取舍

- **快捷键冲突或注册失败**
  - 处理：启动时检测注册结果；失败后在设置页和托盘入口中给出明确提示，并允许用户修改快捷键后重试注册。
  - 取舍：MVP 不做复杂冲突诊断，仅做“失败提示 + 修改入口 + 重试注册”。
- **定时提醒不触发**
  - 处理：提醒服务在启动、设置保存、系统唤醒、跨日切换时重算；汇总窗口也保留手动打开入口。
  - 取舍：MVP 不依赖外部系统级任务计划器，降低复杂度，但要求应用保持常驻。
- **窗口被遮挡或未正确唤起**
  - 处理：显示窗口时执行显示、聚焦、置顶的统一顺序；失败时托盘闪烁或提示用户从托盘手动打开。
- **数据保存失败**
  - 处理：所有写入使用事务；失败时不清空输入；记录非敏感错误类型。
- **配置缺失**
  - 处理：AI 配置缺失仅影响 AI 润色，不影响基础日报；快捷键和提醒配置有默认值。
- **AI 接口超时/报错/返回非法内容**
  - 处理：统一超时、状态码和解析错误模型；不展示敏感请求细节；始终回退到基础日报。
- **剪贴板复制失败**
  - 处理：显示失败提示，保留文本区原文。
- **当天没有任务**
  - 处理：汇总窗口显示空状态，允许补录任务；若用户仍选择生成，则生成简短空日报提示或空白草稿。
- **托盘常驻增加复杂度**
  - 处理：仅实现最小托盘菜单，不引入复杂状态同步。

## Testing and Validation

### 7. 测试与验证方案

#### 关键流程的手动验收步骤

- 启动应用，确认托盘入口可见，快捷键可正常呼出快速记录窗口。
- 连续录入多条任务，关闭并重新打开应用，确认当日数据仍存在。
- 修改下班时间为当前时间后几分钟，等待提醒触发，确认汇总窗口自动出现。
- 在汇总窗口中修改状态与备注，生成基础日报，确认结果文本正确区分状态。
- 配置有效 AI 参数并执行润色，确认生成内容自然、简洁且不夸大原始信息。
- 清空或破坏 AI 配置后再次点击 AI 润色，确认系统降级为基础日报而不是中断。
- 点击复制并验证剪贴板结果；模拟复制失败时确认失败提示存在。

#### 可以做自动化测试的层面

- 任务新增、状态修改、备注编辑的业务规则。
- 当日任务聚合与日报文本组装。
- 提醒时间计算与重算逻辑。
- AI 请求构造、响应解析和错误映射。
- SQLite 仓储层的基本 CRUD 和迁移测试。

#### 需要重点回归验证的点

- 快捷键修改后重新注册。
- 跨日后当日列表切换与提醒重算。
- 备注修改后再次生成日报是否生效。
- AI 失败后基础结果是否保留。
- 空任务日提醒、打开汇总、生成结果的表现是否稳定。

#### 打包前应确认的行为

- 安装后首次启动能够进入后台常驻。
- 默认快捷键、默认提醒设置和默认模板可用。
- 应用退出后再次启动不会丢失数据库与设置。
- 敏感配置不会出现在控制台日志、错误提示或明文调试输出。
- 在 Windows 11 常见分辨率下，三个窗口内容均可完整显示。

## MVP and Later Phases

### 8. MVP 与后续阶段划分

#### MVP 必做

- Tauri 桌面宿主与常驻后台入口
- 全局快捷键呼出快速记录窗口
- 快速录入并保存任务到本地 SQLite
- 按当天汇总展示任务列表
- 到提醒时间自动弹出汇总窗口
- 编辑任务状态与备注
- 设置页中的快捷键、提醒时间、AI 配置
- 基础日报生成
- AI 润色调用与失败降级
- 一键复制日报文本

#### 第二阶段增强

- 更多任务状态类型
- 可选日报模板
- 更丰富的托盘状态和快捷操作
- 手动选择历史日期查看/补录
- 多 AI 服务配置预设与切换
- 更细粒度的提醒策略

#### 目前不建议实现

- 账号系统与云同步
- 多设备同步
- 多人协作
- 周报、月报、统计分析面板
- 项目/标签/优先级体系
- 审批流、企业系统集成
- 围绕 AI 的复杂参数调优面板
